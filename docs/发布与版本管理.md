# 发布与版本管理

## 本地提交

每完成一轮用户确认的修改，都按以下顺序处理：

1. 检查本轮变更范围，不混入无关文件。
2. 运行相关测试；发布候选必须运行完整测试。
3. 扫描个人信息、密钥、真实项目数据和大型二进制文件。
4. 创建一个说明本轮目的的本地 Git commit。
5. 向用户报告 commit、测试结果和仍存在的限制。

本地 commit 是可回退的版本点，不等同于已经上传 GitHub。

## GitHub 更新门槛

只有用户明确确认当前版本可以上传时，才允许：

- 配置或使用 GitHub 远程仓库。
- 执行 `git push`。
- 创建 tag 或 GitHub Release。
- 上传 Windows 安装程序、便携版和 SHA-256 校验文件。

## 版本号

项目使用语义化版本：

- `PATCH`：兼容性修复和小改进。
- `MINOR`：向后兼容的新工作流或明显功能扩展。
- `MAJOR`：存在不兼容的数据格式或工作流变化。

尚处于用户验证阶段的版本在发布说明中标记为 Beta。Git tag 使用 `v<版本号>`，例如 `v0.3.3`。

## GitHub Release

源代码保存在 Git 仓库中；安装包不进入 Git 历史。每个 Windows Release 至少包含：

- 安装版 EXE。
- 便携版 EXE。
- SHA-256 校验文件。
- 中文发布说明，包含已验证功能、已知限制和数据迁移提示。

用户应从 GitHub 的 Releases 页面下载可运行程序，而不是从 `Code -> Download ZIP` 获取安装包。
