# GeneSnap Workbench 架构说明

GeneSnap Workbench 使用分层但不过度耦合的结构。界面负责收集输入和展示状态，应用层编排工作流，领域层保存稳定规则，序列内核执行可测试的纯计算，外部服务和本地存储分别通过独立模块接入。

## 主要模块

- `app`：PySide6 桌面界面、应用服务和命令编排。
- `domain`：项目、构建、设计版本、测序记录和状态历史等领域模型。
- `sequence_core`：序列标准化、坐标规则、shRNA、表达、reporter 和 SYN 设计逻辑。
- `vector_library`：用户导入载体、插入边界、protocol、校验值和公开 starter。
- `sequencing`：测序文件匹配、序列比较和 PASS/FAIL/WARNING 判定。
- `template_engine`：用户 Excel 模板映射以及 XLSX、DOCX、JSON、GenBank 输出。
- `project_workflow`：项目目录、工作日、状态转换、加测/重做和实验记录。
- `integrations`：NCBI、Broad GPP 等外部服务；网络失败必须转换为可理解的状态，不能伪装为验证通过。
- `storage`：SQLite 和本地配置持久化。

## 关键边界

1. `sequence_core` 不依赖桌面界面或数据库，保证核心设计逻辑可以独立测试。
2. 正式设计保存为带校验值和版本号的记录，后续导出和判读必须引用同一个设计版本。
3. 用户导入的载体、模板和测序数据仅保存在本地用户目录或项目目录，不进入源码仓库。
4. 在线查询结果属于外部输入，必须校验、保留来源并支持失败后的人工复核路径。
5. 软件判定与实验验证分开记录，不能用“测试通过”代替湿实验验证。

## 工作流扩展

工作流通过稳定定义注册。新增工作流应分别提供领域模型、设计服务、持久化、导出和界面入口，并拥有独立测试。CKO/Flox 当前保持禁用状态，只有在完整 SOP 和公开或脱敏的代表性案例准备完成后才进入实现范围。
